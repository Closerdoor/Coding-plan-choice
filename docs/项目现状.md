## 3. 固定执行流程

- 从当前阶段开始，`catalog` 主线按“每完成一个厂商即完成一次闭环”的方式推进。
- 已新增日常提交规则：若某一轮对话产生文件改动，则在该轮结束后必须执行一次本地 git commit，但不 push；若无改动则不创建空 commit。
- 每新增并完成一个厂商套餐接入后，必须依次执行：
  - `python -m catalog.update`
  - `python -m unittest catalog.UTtest.selftest`
- 必须确认以下正式产物已经更新：
  - 根目录 `CATALOG_GENERATED.md`
  - `catalog/output/CATALOG_DATA.json`
- 验证通过后，立即提交并推送到 GitHub。
- 默认按以下双轨流程执行，除非用户明确提出不同要求：
  - 日常轮次：有改动就本地 commit，不 push
  - 套餐闭环轮次：完成验证后 commit 并 push GitHub
