# 项目需求文档

## 1. 项目概述
本项目用于收集与展示市面上的 AI 模型与 coding plan/套餐信息，统一以 Markdown 文档对外呈现，并支持自动更新。

## 2. 项目目标
- 收集与展示：汇总各厂商 AI 模型与 coding plan 套餐的简介与费用。
- 统一格式：每家套餐按固定结构展示，便于横向对比与持续扩展。
- 自动更新：支持按固定频率自动刷新（暂定每周 2 次）。
- 官方来源：关键字段必须标注官方信息源链接，确保可追溯。
- GitHub 托管：项目上传 GitHub；自动更新结果体现在仓库的 Markdown 变更中。

## 3. 展示与交付物
- 主展示文档：CATALOG_GENERATED.md（中文，根目录，自动生成）
- 主线结构化产物：`catalog/output/CATALOG_DATA.json`（自动生成，供版本管理与 Markdown 渲染使用）
- 项目说明：README.md（后续补充，可选）
- 数据与自动化实现方式：后续讨论确定（当前阶段先对齐最终展示效果）

## 3.1 CATALOG_GENERATED.md 展示规范（已确认）
- 当前展示实现按 `catalog/catalog_config.py` 的配置顺序输出厂商条目；暂不额外显示 1..N 编号。
- 每个条目使用统一模板。
- 套餐信息采用“竖排对比”表格：
  - 列：各套餐（例如 lite / pro）。
  - 行：固定 5 行字段（价格、用量、支持模型、支持工具、使用方式）。
- 每个条目顶部固定字段：
  - 官方信息源（URL）
  - 最后更新时间（UTC）

## 3.2 主线实现架构（已确认）
- `catalog/catalog_config.py` 维护厂商配置清单。
- 一条配置对应一个厂商；一个厂商只对应一个具体套餐页面。
- `catalog_config.py` 中的配置顺序，就是 `CATALOG_GENERATED.md` 中的输出顺序。
- 每增加一个厂商配置，就新增一个对应的厂商脚本；因为各厂商官网结构不同，抓取和解析逻辑允许彼此不同。
- 各厂商脚本位于 `catalog/sources/`，职责是：
  - 优先访问官方地址页或官方 API
  - 当官方地址页描述不清晰，或无法稳定获取目标字段时，再使用说明文档或补充来源进行补充
  - 抓取并解析套餐字段
  - 返回统一结构的厂商数据
- 总入口脚本为 `catalog/update.py`，职责是：
  - 按配置顺序调用各厂商脚本
  - 汇总所有厂商的结构化数据
  - 生成 `catalog/output/CATALOG_DATA.json`
  - 基于该 JSON 渲染根目录 `CATALOG_GENERATED.md`

## 3.3 主线数据流（已确认）
- 子脚本不直接生成 Markdown。
- 子脚本优先返回统一结构化数据，由 `catalog/update.py` 统一写出 JSON 与 Markdown。
- `catalog/output/CATALOG_DATA.json` 是主线结构化数据产物，也是版本管理中的主要比对对象。
- 根目录 `CATALOG_GENERATED.md` 是展示产物，由 JSON 统一渲染得到。
- 后续如果套餐价格、额度、支持模型或支持工具发生变化，优先通过 Git 对 `CATALOG_DATA.json` 与 `CATALOG_GENERATED.md` 的变更进行追踪。

## 3.4 主线 JSON 结构策略（已确认）
- `CATALOG_DATA.json` 只保留一份，汇总全部厂商数据，不按厂商拆分为多个 JSON 文件。
- 当前不强行固定完整 schema，而是采用“公共骨架 + 可扩展字段”的方式推进。
- 当前已确认的公共骨架：
  - 厂商级字段：`vendor_id`、`company_name`、`vendor_name`、`official_sources`、`updated_at_utc`、`packages`
  - 套餐级核心字段：`name`、`price`、`quota`、`models_raw`、`models_filtered`、`tools`、`access_method`
- 当不同厂商页面结构差异较大、暂时无法统一时，允许在后续实现中增加补充字段，但不能破坏公共骨架。

## 3.5 支持模型双字段策略（已确认）
- 每个套餐的“支持模型”在 JSON 中保留两份：
  - `models_raw`：从厂商官网直接解析到的原始模型列表
  - `models_filtered`：基于 Top3 过滤后的模型列表
- `CATALOG_GENERATED.md` 渲染时，默认优先展示 `models_filtered`。
- 若 `models_filtered` 为空，则退化为展示 `models_raw`。
- 这样做的目的是保留“官网原始信息”和“展示过滤结果”两份数据，便于版本管理、问题排查与后续过滤逻辑调整。

## 4. 收录范围（Top10 的方向）
- 中国大陆厂商 coding plan：腾讯、阿里、字节、百度。
- 垂直大模型厂商套餐：后续按清单补充。
- 美国厂商的模型/套餐：后续按清单补充。
- 产品形态：模型 API/推理计费、订阅套餐、开发者工具订阅、企业套餐等（以官方口径为准）。

## 5. 内容规范（每家/每套餐必须包含）
- 官方地址（产品页）
- 信息源（官方链接，至少覆盖价格/用量/支持模型/支持工具/使用方式等字段）
- 价格（币种按厂商原始币种：CN=CNY，US=USD）
- 用量/额度（按官方口径原样表达，如请求次数、token、席位、时间窗口等）
- 支持模型（名称列表，或指向官方支持列表页）
- 主流编程工具（按官方口径）
- 使用方式（如 API Key、插件登录、企业开通等，按官方口径）
- 最后更新时间（UTC；由自动更新流程写入）

## 5.5 套餐周期与价格展示规则（已确认）
- 某个厂商的 Coding Plan 在官方页面中，可能同时存在按月、按季度、按年等多种付费周期。
- 各厂商脚本可以抓取并保留这些不同周期的套餐信息，供后续结构化扩展或规则调整使用。
- 但在当前阶段，`CATALOG_GENERATED.md` 的正式展示只输出按月付费价格。
- 因此当同一套餐存在多种周期时，当前主展示口径统一为月付价格，例如 `¥40/月`。
- 若官方页面仅提供季度价或年价，且无法明确换算出月付价格，则应如实记录该情况，并在实现时单独处理，不能擅自折算伪造月价。
- 后续若需要扩展季度价、年价展示，再基于已抓取的结构化数据调整渲染规则；当前不要在 Markdown 主展示中并列输出多周期价格。

## 5.2 信息源优先级规则（已确认）
- 在每一个厂商脚本中，必须优先使用官方地址页作为主信息源。
- 只有当官方地址页对目标字段描述不清晰，或页面结构导致无法稳定获取目标字段时，才允许使用说明文档或补充来源进行补充。
- 这样做的原因是：官方地址通常更稳定，而说明文档链接和正文结构更容易变动。
- 因此后续开发中，不允许默认优先抓说明文档；必须先尝试官方地址，再按需补充说明文档。

## 5.3 用量与计费方式展示规则（已确认）
- 当前套餐用量的计费方式，统一按以下 3 类口径识别与展示：
  - prompt 次数
  - Token 成本计费
  - 按请求数计费
- 每个厂商脚本都必须从官方套餐信息中提取明确的用量内容，并按实际口径输出；不能用模糊文案替代明确额度。
- 若官方页面给出的是绝对额度，则必须在 JSON 与 Markdown 中展示明确内容，例如：
  - `80次prompts/5小时；400次prompts/周`
  - `1200次请求/5小时；9000次请求/周；18000次请求/月`
- 若官方页面给出的是 Token 成本计费，则必须明确展示 Token 或成本口径，而不是混写成请求数或 prompt 次数。
- 若官方地址未写清楚用量，但说明文档中存在明确额度，可按“官方地址优先，说明文档补充”的规则补全后展示。
- 后续开发与修正中，必须优先保证“用量字段可读、明确、可比较”，避免再次输出模糊描述。

## 5.4 使用方式采集与展示规则（已确认）
- `使用方式` 字段必须优先依据厂商官方地址、官方快速开始文档或官方工具接入文档采集，不允许继续使用无依据的硬编码占位值。
- 采集时至少确认以下 3 个要素是否存在：
  - 是否提供 API Key
  - 是否要求配置 Base URL
  - 是否区分 OpenAI / Anthropic 协议
- 当官方资料明确表明套餐通过 API Key + Base URL 接入，且同时提供 OpenAI / Anthropic 两种兼容协议时，展示层统一采用短版文案：
  - `API Key + Base URL（OpenAI / Anthropic 协议）`
- 若后续某厂商官方口径明显不同于该通用模式，例如插件登录、企业开通、账号授权等，则必须按官方实际口径单独展示，不能强行套用该短版文案。
- 若官方地址页未写清楚 `使用方式`，可按“官方地址优先、说明文档补充”的规则，从官方说明文档或工具接入文档补充确认后再展示。

## 5.1 支持模型展示策略（已确认）
- 套餐“支持模型”字段以官方信息源页面为准进行抓取与解析。
- 展示时默认进行过滤：仅展示为对应“模型提供商”生成的 Top3 模型清单中命中的模型。
- 退化策略：若过滤后为空（与 Top3 清单无交集），则退化为展示官方信息源中解析到的模型内容。
- 模型命名以官方写法为准；允许维护一份别名表用于匹配（避免大小写、符号、别名导致无法命中）。
- Top3 过滤逻辑统一放在 `catalog/update.py` 中完成，而不是分散写在各厂商子脚本中。
- 各厂商子脚本只负责解析官网原始模型列表，并返回 `models_raw` 所需数据。

## 6. 自动更新要求（第一阶段）
- 更新频率：每周 2 次（使用 UTC 配置）。
- 最小更新定义：
  - 校验官方来源页面；
  - 刷新 `catalog/output/CATALOG_DATA.json`；
  - 刷新“最后更新时间（UTC）”；
  - 若官方信息发生变化，应同步更新 CATALOG_GENERATED.md 的对应内容。
- 更新结果：当 CATALOG_GENERATED.md 发生变化时，自动提交到 GitHub 仓库。

## 6.5 提交流程规则（已确认）
- 若某一轮对话产生了文件改动，则在该轮输出结束后，必须将本轮改动记录为一次本地 git commit，但不 push。
- 若某一轮对话没有任何文件改动，则不创建空 commit。
- 该本地 commit 规则适用于日常迭代、规则调整、文档修订和脚本修改等所有实际改动场景。
- 同时保留既有套餐闭环规则不变：当完成一个套餐接入或完成一轮套餐级改动闭环后，仍需按原流程执行生成、测试、检查产物，并提交后推送到 GitHub。
- 因此后续执行时应区分两类提交：
  - 日常轮次：本地 commit，不 push
  - 套餐闭环轮次：完成验证后 commit 并 push GitHub

## 6.4 主线失败策略（已确认）
- 单个厂商抓取/解析失败时，不覆盖该厂商在 `CATALOG_DATA.json` 中的旧数据。
- 单个厂商失败不应导致整个 `catalog.update` 直接中断；总脚本应尽可能继续处理其它厂商。
- 若某厂商是首次接入且本次抓取失败、仓库中又不存在该厂商旧数据，则该厂商本次可暂时缺席输出，并记录 warning。

## 6.1 自动更新范围（主干与支线）
- 主干（套餐信息）：从各条目的官方信息源 URL 抓取/解析套餐价格、用量、支持工具、使用方式等，先更新 `catalog/output/CATALOG_DATA.json`，再生成根目录的 `CATALOG_GENERATED.md`。
- 支线（Top3 清单）：从多个公认榜单/评测汇总生成“模型提供商 Top3 模型清单”，并用于过滤 CATALOG_GENERATED.md 的“支持模型”展示（无交集时退化展示官方内容）。

## 6.3 主线增量开发方式（已确认）
- 主线 `catalog` 按厂商逐个增量实现。
- 每次新增一个厂商时，至少同步完成以下内容：
  - 在 `catalog/catalog_config.py` 中新增该厂商配置
  - 在 `catalog/sources/` 中新增对应抓取/解析脚本
  - 让总入口 `catalog/update.py` 可以将该厂商纳入统一 JSON 和 Markdown 产物
- 每完成一个厂商后，默认必须继续执行完整闭环：
  - 运行 `python -m catalog.update`
  - 运行 `python -m unittest catalog.UTtest.selftest`
  - 确认 `CATALOG_GENERATED.md` 与 `catalog/output/CATALOG_DATA.json` 已更新
  - 验证通过后，立即提交并推送 GitHub
- 主线能力以“每增加一个厂商，就能真实进入正式产物”为完成标准，而不是仅补充样例文档。

## 6.2 支线产物与运行方式（已落地）
- 一键更新命令（会触网抓取榜单）：`python -m rankings.update`
- 单元测试（不触网）：`python -m unittest rankings.UTtest.test_rankings`
- 产物：
  - `rankings/output/MODEL_RANKING.md`：模型能力榜单（自动生成）
  - `rankings/output/MODEL_SCORES.json`：结构化中间产物（供 Top3 生成与主线过滤使用）
  - `rankings/output/MODEL_TOP3.md`：按模型原厂 Top3 清单（自动生成）
- 失败策略：
  - 单个榜单源抓取/解析失败将被跳过，并在产物的 warnings/告警中记录失败信息。
  - 若本次执行整体失败（抛异常/非 0 退出）或聚合结果为 0 个模型，则保留上次成功产物，不覆盖 `rankings/output/`。

## 7. 验收标准（第一阶段）
- 根目录存在 CATALOG_GENERATED.md；docs/ 下存在本需求文档。
- CATALOG_GENERATED.md 至少包含“腾讯云大模型 Coding Plan”条目，字段齐全、表格列固定、信息源为官方链接。
- 后续新增第二家厂商时，不需要改动版式，只需要按同样结构补充条目。
