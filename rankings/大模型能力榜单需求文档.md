# 大模型能力榜单需求文档（rankings）

目标：从多个公认度较高的模型评测/榜单抓取数据，聚合生成一份“模型能力榜单”，并进一步生成每个【模型原厂/Provider】的 Top3 模型清单。

本模块是项目的支线能力组件，产物用于：
- 人工审阅与追溯（排名表、告警）
- 主线 catalog 在展示“支持模型”时做筛选（只展示命中 Provider Top3 的模型；无命中则退化展示官方解析列表）

---

## 1. 背景与动机
- 主展示需要“支持模型”字段。
- 官方信息源往往列出很多模型，全部展示会降低可读性。
- 需要一个相对客观、可更新的方式选出“代表性强模型”（每个模型提供商 3 个）。

---

## 2. 输入与输出

输入：
- 多个榜单/评测站点的公开数据（第三方）。

说明：本模块不依赖主线的“官方可用模型列表”。主线如果需要做二次过滤，会基于本模块产物自行处理。

输出：
输出（固定三件套，位于 `rankings/output/`）：
- `MODEL_RANKING.md`：全局模型能力榜单（markdown，含告警）
- `MODEL_SCORES.json`：结构化中间产物（供主线读取）
- `MODEL_TOP3.md`：按 Provider 分组的 Top3（markdown，含告警）

约束（已确认）：
- 模型标准名称以官方写法为准。
- 允许维护别名表用于匹配。
- 若 Top3 与官方可用模型无交集，主展示退化为展示官方解析到的模型内容。

范围边界（已确认）：
- 本支线只关心“模型原厂/Provider”的模型强弱与 Top3 选取。
- 不将“提交组织/团队/聚合商”视为 Provider；仅将其作为原始数据字段保留（如 SWE-bench 的 Org）。

---

## 3. 数据源（建议组合）

说明：没有单一“唯一权威”榜单，建议至少覆盖：
- 通用对话/综合能力（偏主流模型覆盖、更新快）
- 代码能力（更贴近 coding plan 场景）
- （可选）中文能力（覆盖国内模型生态）

当前已接入的数据源（实现位于 `rankings/sources/`）：
- LMSYS LLM Arena（Chatbot Arena）
  - URL: https://lmarena.ai/leaderboard
  - 特征：人类偏好对战 Elo，更新较快。
- Stanford CRFM HELM
  - URL: https://crfm.stanford.edu/helm/latest/
  - 特征：学术机构，多维度评测，方法透明。
- SWE-bench（代码能力）
  - URL: https://www.swebench.com/
- LiveCodeBench（代码能力）
  - URL: https://livecodebench.github.io/

备注：
- HELM 当前为占位源（`rankings/sources/helm.py` 返回空列表），用于保留稳定的聚合接口与权重口径；后续如接入真实数据源，直接替换其 `fetch()` 实现。

备注：
- “官方渠道”要求适用于主干套餐信息抓取；榜单是第三方，仅用于能力排序与选型依据。

---

## 4. 聚合口径（需要讨论并最终确定）

### 4.1 分榜 vs 总榜
建议至少两条线：
- 通用能力榜（聊天/综合）
- 代码能力榜（coding 相关）

Top3 生成策略（已确认）：代码榜单优先。

实现方式：综合 5 个数据源，但让“代码能力”来源权重更高。

### 4.2 评分融合
不同榜单指标不同（Elo、准确率、综合分等），需要做标准化：
- 标准化策略：将每个榜单内的名次转换为 0..1 的 unit score（按排名分位数），再按权重加权。

实现细节：
- 对每个数据源：同一 (model, provider) 取最大 raw score 后排序，按名次转换 unit score。
- 跨源聚合：只对“该模型出现过的源”做加权平均，缺失源不计入分母。

权重（建议默认，代码优先且尽量均衡）：
- SWE-bench：0.30
- LiveCodeBench：0.30
- LMSYS LLM Arena：0.15
- Stanford HELM：0.15
- OpenCompass：0.10

缺失处理（建议默认）：
- 某模型未出现在某个数据源：该源不计入分母（只对“出现过的源”做加权平均）。
- 为防止“只出现在单一榜单就冲到前排”，可设置最小覆盖要求（例如至少命中 2 个数据源才可进入 TopK）；是否启用后续再定。

### 4.3 过滤规则
本模块的过滤规则（对所有源与所有输出统一生效）：
- 排除蒸馏/量化/推理格式与“小号”关键词：`distill/quant/int4/int8/4bit/8bit/gguf/awq/gptq/mini/small/lite/tiny`
- 排除部分明显非 LLM 文生文模型关键词：`image/video/t2v/i2v/flux/sora/...`（见 `rankings/core.py` 的 `EXCLUDE_NAME_PATTERNS`）
- 参数量阈值：若模型标识可解析 `xxB`，则 `<70B` 视为小模型并排除；MoE 形如 `235B-A22B` 取总参数量 `235B`。

说明：主线的“官方可用模型列表”交集过滤不在本模块内实现，属于 catalog 侧逻辑。

### 4.4 退化策略（已确认）
- 若 Top3 过滤后为空，则主展示直接展示官方信息源解析到的模型内容。

---

## 5. 生成流程（端到端）
1. 拉取榜单数据
  - 优先使用榜单提供的结构化数据（API/JSON/CSV）。
  - 若仅有网页：抓取 HTML 并解析。
2. 抽取并规范化模型名
  - 使用“官方写法优先 + alias 表补充”进行统一。
3. 生成全局模型能力分数/排名（可选）
  - 保存为一份可追溯的榜单产物（例如 markdown 或 json）。
4. 为每个模型提供商生成 Top3 清单
  - 先按提供商聚合。
  - 过滤掉蒸馏/量化/小模型。
  - 取 Top3；不足 3 个则按实际数量。
5. 输出用于主干展示
  - 主线渲染时读取 `rankings/output/MODEL_SCORES.json` 或 `MODEL_TOP3.md` 来过滤“支持模型”。

---

## 6. 更新频率与任务拆分
- 建议榜单生成频率：每周 1 次（或与主干同频，每周 2 次）。
- 建议与主干拆分为两个步骤：
  - Step 1：生成/更新 Top3 清单
  - Step 2：抓取官方信息源并更新 CATALOG.md（依赖 Step 1 的结果做过滤；无结果则退化展示官方）

---

## 7. 风险与应对
- 网络/访问失败：
  - 策略：保留上次成功结果；本次失败不覆盖。
- 页面结构变化导致解析失败：
  - 策略：优先找结构化端点；解析器做容错；失败回退到上次结果。
- 榜单口径差异大：
  - 策略：采用“分位数标准化 + 权重”融合，避免直接拼不同分数。
- 模型重命名/别名过多：
  - 策略：维护 alias 表；并输出“未匹配模型列表”供人工补别名。

## 9. 当前实现状态（已落地）

本方案已实现为一套 **Python 标准库（stdlib-only）** 脚本，可一键更新产物，并支持后续增删榜单数据源。

### 9.1 一键更新命令
- 入口命令：`python -m rankings.update`
- 执行流程：
  - Step 1（唯一触网步骤）：抓取 + 聚合，生成 `rankings/output/MODEL_RANKING.md` 与 `rankings/output/MODEL_SCORES.json`
  - Step 2（不触网）：读取 `MODEL_SCORES.json`，生成 `rankings/output/MODEL_TOP3.md`

### 9.2 产物说明
- `rankings/output/MODEL_RANKING.md`：全局模型能力榜（按 unit score 加权平均；缺失数据源不计入分母）
- `rankings/output/MODEL_SCORES.json`：结构化中间产物（包含 weights、filters、warnings、models 明细与 per-source unit scores），用于 Top3 生成与主线过滤
- `rankings/output/MODEL_TOP3.md`：按 Provider 分组的 Top3 清单（用于主线“支持模型”展示过滤的输入之一）

### 9.3 代码结构（便于增删榜单）
- 核心逻辑：`rankings/core.py`
  - provider 归属推断与白名单回退
  - 小模型过滤（关键词 + 参数量阈值）
  - 跨源 join/dedupe、分位数标准化、加权聚合
  - Arena 噪声门控（仅保留在非 Arena 源也出现过的 Arena 模型）
- 数据源模块（每个榜单一个文件，包含抓取与解析）：`rankings/sources/*.py`
  - `livecodebench.py`、`swebench.py`、`arena.py`、`opencompass.py`、`helm.py`（占位，当前返回空）
- 数据源注册表：`rankings/sources/__init__.py` 的 `SOURCES = [...]`
  - 后续新增/删除榜单：只需要新增/删除对应 source 模块，并在 `SOURCES` 里注册/移除即可

### 9.4 已固化口径（实现与文档一致）
- 权重：SWE-bench 0.30、LiveCodeBench 0.30、Arena 0.15、HELM 0.15（占位）、OpenCompass 0.10
- 缺失处理：模型未出现在某个数据源时，该源不计入分母（只对“出现过的源”做加权平均）
- 小模型过滤（对所有数据源、所有输出统一生效）：
  - 命名关键词：distill/quant/int4/int8/gguf/awq/gptq + mini/small/lite/tiny 等
  - 参数量阈值：若模型名可解析到 `xxB`（含 MoE 形如 `235B-A22B` 取总参数量），则 `<70B` 视为小模型并排除
- provider（模型原厂）归属：
  - 优先从模型标识（含 HuggingFace `org/repo` 或 URL）推断原厂，避免把提交组织/聚合商当作模型原厂
  - 最后回退到源的 org/provider 字段时，仅允许落入“已知原厂白名单”，否则标记为 `Unknown`
  - Top3 输出跳过 `Unknown`

### 9.5 失败策略
- 单个榜单抓取/解析失败：跳过该源，并将失败信息记录到：
  - `rankings/output/MODEL_RANKING.md` 的“抓取/解析告警”
  - `rankings/output/MODEL_SCORES.json` 的 `warnings`
